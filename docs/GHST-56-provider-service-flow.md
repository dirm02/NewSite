# GHST-56 — Provider service card creation flow (audit + fix)

Audited the Add/Edit Service flow end to end and fixed the bugs that prevented a
provider from actually creating a real `services` record. Service cards across
the app are generated by the frontend from PocketBase `services` + expanded
`provider_profiles`/`category`/`city` — never hand-authored static cards.

## End-to-end flow (after fix)

```
Provider sidebar → My Services (ProviderServicesPanel, live)
  → "Add Service" / per-card "Edit"  → /providers/provider-edit-service[?id=]
     → ProviderEditService → ProviderServiceFormPanel
        → useProviderServiceActions.createService/updateService
           → jobs.ts createService/updateService (multipart)
              → POST/PATCH /api/collections/services (PB create/update rules)
  → back to My Services (live list re-reads provider's services)
Public discovery (home / grid / list / search / details) reads the same
`services` records (status='active') → cards generated from real data.
```

## Bugs found (verified by code + production browser + API) and fixes

| # | Problem | Effect | Fix |
|---|---|---|---|
| 1 | `providerEditService` route (`/providers/provider-edit-service`) was **never registered** in `router.link.tsx` | "Add Service" / "Edit" CTAs → **Error404** (verified on prod). Provider could not create/edit a service at all. | Registered `ProviderEditService` in `providerRoutes`. |
| 2 | `createService` never set the **required, unique** `slug` | Create would fail PB validation even if the route worked | `buildServiceSlug(title)` = `slugify(title)-<rand6>` (guarantees uniqueness without a pre-check round-trip) |
| 3 | `category` (a **required** relation) was passed as `undefined` when blank, and the form allowed empty category | Create would fail PB validation; honest UX missing | Form marks Category required + validates; API only sends category when present |
| 4 | No image upload (`services.images`), though the field exists and GHST-56 requires "uploads/chooses images" | Cards always fell back to a placeholder; providers could not add photos | Added multipart upload: `pbCreateForm`/`pbUpdateForm` (client), `images?: File[]` on `ServiceFormInput`, and a file input (JPG/PNG/WebP, up to 10) in the form |

## PB `services` schema contract (authoritative — from migration `1740523500`)

| Field | Type | Required | Notes |
|---|---|---|---|
| `title` | text (1–200) | ✅ | |
| `slug` | text (1–200) | ✅ **unique** | now generated client-side |
| `category` | relation→service_categories | ✅ | now required in the form |
| `provider` | relation→provider_profiles | ✅ | set from auth provider profile |
| `status` | select active/pending/inactive | ✅ | created `active` (MVP, no admin approval yet) |
| `description` | editor | — | |
| `price_from` / `price_to` | number ≥0 | — | CAD |
| `duration_minutes` | number ≥0 | — | |
| `images` | file ×10, jpg/png/webp, 5 MB | — | now uploadable |
| `city` | relation→cities | — | Canada-only list |
| `featured` | bool | — | not provider-editable (admin/marketing) |
| `rating_avg` | number | — | derived; not provider-editable |

**Access rules:** create requires `auth.role='provider' && provider.user = auth.id`;
update/delete `provider.user = auth.id || admin`. The form/API satisfy these
(provider id resolved from the signed-in user's profile).

  provider, creates a service via multipart (generated slug + required category),
  asserts `status=active` and correct provider/category, then **deletes it**
  (HTTP 204) so no residue is left in production. (Confirms the create contract
  works against live PB rules.)
- `node scripts/audit-broken-routes.mjs` → `providerEditService` no longer listed
  (broken-route count 17 → 16; the remaining 16 are non-MVP demo/alt-template).
- `npm run build` → **PASS**. Lint clean on changed files.
- Not deployed; PocketBase not restarted; no collections created/deleted.

## Remaining gaps / recommendations (future tasks)

- **Status moderation:** services are created `active` immediately (no admin
  approval). When an admin moderation phase exists, default new services to
  `pending` and discovery (already `status='active'`) will exclude them.
- **Edit-mode image management:** new uploads *replace* the image set (PB
  `images`); fine-grained add/remove of individual existing images is deferred
  (would use PB `images+`/`images-` modifiers).
- **Provider aggregate fields** (`provider_profiles.services_count`) are not
  recomputed on create/delete — see GHST-55 (data honesty); recommend a hook or
  computed read instead of the seeded value.
- **Slug readability:** generated slugs carry a random suffix for guaranteed
  uniqueness; a nicer scheme (collision-checked clean slug) can replace it later.
